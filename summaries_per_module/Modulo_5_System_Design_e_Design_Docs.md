# Estudo sobre System Design

## Introdução
- O System Design é um tema prático e essencial para entender a definição de arquitetura de sistemas.
- Importante ter uma base teórica antes de partir para a prática.

## O que é System Design?
- *Definição*: Processo de definição de arquitetura, componentes, módulos, interfaces e dados para atender requisitos especificados.
- *Importância*: Vai além do simples desenho de arquitetura; envolve um processo intencional de resolução de problemas.

## Importância do System Design
- Ajuda a:
  - Racionalizar definições importantes.
  - Seguir etapas no processo de design, evitando decisões precipitadas.
  - Explorar diversas soluções para um problema específico.
- O resultado não é um documento com todos os detalhes, mas aproximações que ajudam a entender a arquitetura.

## Etapas do System Design
1. **Definição da Arquitetura**: Compreender que o design é um processo, não uma tarefa única.
2. **Análise do Problema**: Identificar as necessidades e desafios que devem ser resolvidos.
3. **Criação de Planos de Capacidade**: Planejar como o sistema escalará conforme as necessidades aumentam.
4. **Modelagem de Dados**: Estruturar como os dados serão organizados e acessados.
5. **Aplicação Prática**: Colocar em prática os conceitos discutidos.

## Exploração de Soluções
- No processo de system design, é comum encontrar várias soluções para um único problema.
- A escolha da solução deve considerar:
  - Custo financeiro (ex: uso de cache em memória vs. outras abordagens).
  - Desempenho e eficiência.
  - Implicações a longo prazo (ex: armazenamento).

## Considerações Finais
- O design de sistemas não é apenas sobre criar desenhos, mas sobre pensar em soluções práticas e eficazes.
- É fundamental exercitar a mente para planejar e avaliar diferentes abordagens.
- No próximo tópico, será abordado o *System Design em relação às grandes empresas de tecnologia* (Big Techs).

## Dicas para Estudo
- **Revisar os conceitos**: Aprofundar-se em cada etapa do System Design.
- **Praticar**: Tentar aplicar os conceitos em projetos reais ou hipotéticos.
- **Discutir**: Participar de discussões sobre soluções e abordagens de design com colegas.

## Conclusão
O System Design é uma habilidade crítica para profissionais de tecnologia, que envolve não apenas o desenho, mas uma compreensão profunda das necessidades e soluções de arquitetura. É um processo que requer prática e reflexão constante.

  - Também dividir por \( 10^3 \).

# Estudo sobre Storage

## Introdução
- A seção de Storage é considerada uma das partes menos interessantes, mas essencial para entender a gestão de dados.
- O cálculo de Storage é simples, mas requer atenção aos detalhes.

## Como Calcular Storage
- **Fórmula Básica**:
  - Storage = (Quantidade de Writes) x (Request Size)
- **Writes Per Second (RPS)**:
  - A quantidade de gravações por segundo que o sistema realiza.

### Replication Factor
- O que é? 
  - É a quantidade de réplicas que você mantém de um dado.
- Importância:
  - Evita a perda de dados em caso de falhas (ex: HD queimado).
  - Em sistemas como Apache Kafka, o Replication Factor é utilizado para duplicar partições.
  
### Cálculo do Storage com Replication Factor
- **Fórmula Ajustada**:
  - Storage = (Quantidade de Writes) x (Request Size) x (Replication Factor)

## Exemplos Práticos
- Exemplo de cálculo:
  - RPS: 5
  - Request Size: 50 Kbytes
  - Replication Factor: 3
  - **Cálculo**:
    - Storage = 5 RPS x 50 Kbytes x 3 = 750 Kbytes por segundo.
    - Isso resulta em 0.75 Megas por segundo.

## Cálculo de Storage por Período
1. **Storage por Dia**:
   - Fórmula:
     - Storage por dia = Storage por segundo x 10^5 (100.000 segundos)
2. **Storage por Ano**:
   - Fórmula:
     - Storage por ano = Storage por segundo x 10^5 x 3.65 (dias em um ano)
     - Isso resulta em 3.65 x 10^7.
3. **Storage por 5 Anos**:
   - Fórmula:
     - Storage por 5 anos = Storage por segundo x 3.65 x 10^7 x 5
     - Resulta em 18.25 x 10^7.
   - Para simplificar:
     - Arredondar para 1.8 x 10^8.
     - Arredondar ainda mais para 2 x 10^8.

## Dicas para Arredondamento
- Ao fazer cálculos de Storage, é comum arredondar valores para facilitar a compreensão e a comunicação.
- Lembre-se:
  - É importante entender o raciocínio por trás dos cálculos, mais do que decorar fórmulas.

## Conclusão
- O estudo do Storage é crucial para a gestão eficiente de dados em um sistema.
- Compreender os conceitos de Writes, Request Size e Replication Factor ajuda a prever a necessidade de armazenamento a curto e longo prazo.
- No próximo vídeo, serão realizados cálculos práticos para diferentes períodos (1 dia, 1 ano, 5 anos).

- Praticar os exemplos apresentados.

## Estudo sobre Storage

### Introdução
- A parte de **Storage** é considerada uma das mais complexas, mas não necessariamente difícil.
- O foco principal é como calcular o armazenamento necessário para dados.

### Cálculo de Storage
1. **Fórmula Básica de Cálculo**
   - O cálculo de Storage é feito pela fórmula:
     - **Storage = (Quantidade de Writes) x (Request Size)**
   - Onde:
     - *Writes*: Número de gravações.
     - *Request Size*: Tamanho de cada gravação.

2. **Consideração do Replication Factor**
   - O *Replication Factor* é a quantidade de réplicas que um dado terá.
   - Exemplo:
     - Em um sistema como Apache Kafka, você pode definir o *Replication Factor* ao criar uma partição.
   - A fórmula ajustada considerando o *Replication Factor*:
     - **Storage = (Quantidade de Writes) x (Request Size) x (Replication Factor)**

3. **Importância do Replication Factor**
   - Um *Replication Factor* de 1 significa que haverá apenas uma cópia dos dados.
   - A decisão do *Replication Factor* depende da criticidade dos dados.
   - Exemplo:
     - Para logs, pode não ser necessário um *Replication Factor* alto.

### Cálculo de Storage por Dia, Ano e 5 Anos
1. **Cálculo de Storage por Dia**
   - **Storage por Dia = (Storage por Segundo) x (100.000 segundos)**
     - *100.000 segundos* é uma aproximação de um dia.

2. **Cálculo de Storage por Ano**
   - **Storage por Ano = (Storage por Segundo) x (100.000 segundos) x (3,65)**
   - Onde *3,65* é uma aproximação de dias em um ano.

3. **Cálculo de Storage para 5 Anos**
   - **Storage em 5 Anos = (Storage por Segundo) x (3,65) x (5)**
   - Para facilitar os cálculos, pode-se arredondar para:
     - **Storage em 5 Anos ≈ 2 x 10^8**

### Dicas para Cálculo Rápido
- Realizar aproximações é útil para simplificar cálculos.

# Notas de Estudo sobre Cálculo de Storage

## Introdução
O cálculo de storage pode ser considerado um dos tópicos mais enfadonhos, mas é essencial para entender o gerenciamento de dados em sistemas. Vamos explorar como calcular o storage, levando em conta o *Replication Factor* e fazendo estimativas para diferentes períodos.

## Cálculo Básico de Storage
- **Fórmula Básica:**
  - O cálculo de storage é feito através da fórmula:
    - **Storage = Writes x Request Size**
  - Onde:
    - **Writes**: número de gravações.
    - **Request Size**: tamanho da solicitação em bytes.

## Replication Factor
- O *Replication Factor* é crucial para garantir a integridade dos dados.
  - **Definição**: número de cópias de um dado que você mantém.
  - Exemplo: em sistemas como Apache Kafka, você define o *Replication Factor* ao criar partições.
- **Importância**: Um *Replication Factor* de 1 significa que você tem apenas uma cópia do dado; se essa cópia for perdida, os dados são perdidos.

## Cálculo do Storage com Replication Factor
- **Fórmula Ajustada:**
  - Para incluir o *Replication Factor*, a fórmula se torna:
    - **Storage = Writes x Request Size x Replication Factor**
  - Exemplo prático:
    - Se você tem 5 RPS (Writes Per Second) e um *Request Size* de 50 Kbytes, com um *Replication Factor* de 3:
      - **Cálculo**:
        - 5 RPS x 50 Kbytes x 3 = 750 Kbytes por segundo.

## Estimativas de Storage
### Cálculo Diário
- **Fórmula**:
  - **Storage por Dia = Storage por Segundo x 100.000 segundos**
- **Exemplo**: 
  - Para um storage de 0.75 Megas por segundo:
    - 0.75 MB/s x 100.000 s = 75.000 MB/dia ou 75 GB/dia.

### Cálculo Anual
- **Fórmula**:
  - **Storage por Ano = Storage por Segundo x 100.000 segundos x 365 dias**
- **Exemplo**:
  - Para o mesmo storage:
    - 0.75 MB/s x 100.000 s x 365 = 27.000.000 MB/ano ou 27 TB/ano.

### Cálculo para 5 Anos
- **Fórmula**:
  - **Storage por 5 Anos = Storage por Segundo x 100.000 segundos x 365 dias x 5 anos**
- **Exemplo de Cálculo**:
  - Para 5 anos:
    - 0.75 MB/s x 100.000 s x 365 x 5 = 135 TB.
    - Em termos de arredondamento, pode-se simplificar para cerca de **2 x 10^8 MB**.

## Arredondamento e Estimativas
- O arredondamento é uma prática comum para simplificar cálculos:
  - Por exemplo, 18.25 pode ser arredondado para 1.8 x 10^8.
  - Para facilitar, o storage em 5 anos pode ser estimado como **2 x 10^8**.

## Introdução
- A aula anterior focou em como melhorar a modelagem de sistemas que utilizam *message brokers*.

- A geração de códigos únicos para ingressos pode ser custosa em sistemas de alto throughput.
- O processo envolve várias etapas:
  1. Gerar um código.
  2. Consultar o banco de dados para verificar se o código já existe.
  3. Se o código não existir, registrá-lo e gerar um QR Code.
- Essa sequência pode causar lentidão no sistema.

## Solução Proposta
- **Sistema de Geração Antecipada de Códigos e QR Codes**
  - Em vez de gerar códigos em tempo real, pré-gerar códigos e QR Codes.
  - Um gerador de códigos pode ser implementado para criar códigos em intervalos regulares.
  - Quando um código é necessário, o sistema simplesmente o retira de um conjunto de códigos já gerados.

### Benefícios da Abordagem
- Redução da lentidão na geração de códigos e QR Codes.
- O sistema pode manter um microserviço que associa o código gerado com o ID da ordem.
- A consulta ao banco de dados se torna mais simples, apenas verificando se o código já está em uso.

## Implementação Prática
- **Armazenamento na Nuvem**
  - Os códigos e QR Codes podem ser armazenados em serviços de armazenamento em nuvem como Amazon S3, Azure Blob, etc.
  - Isso facilita a recuperação dos códigos e QR Codes quando necessário.

## Considerações Finais

- A rede é um recurso complexo que deve ser tratado com cuidado.
- **Latência** é um fator crítico; quanto maior o número de **round trips**, maior a latência.
  - Exemplo: Transferência de 1 kilobyte em uma rede de 1 gigabit por segundo leva 10 microsegundos; com múltiplos round trips, isso pode aumentar para 500 microsegundos.

### Cenários de Rede
- Latência aumenta significativamente quando recursos estão **distribuídos** em diferentes regiões.
- Importância de considerar o ambiente em que a rede opera.

## Premissas de Design
### 1. Compressão de Dados
- Dados devem ser **comprimidos** para otimizar o uso da rede e melhorar o tempo de resposta.

### 2. Armazenamento
- O armazenamento tem sido um ponto crítico para o tempo de resposta em aplicações.
- Comparação entre **HDD** e **RAM**:
  - HDD: 7.200 RPM (rondas por minuto) impacta o tempo de leitura.
  - A leitura de dados em HDD é mais lenta do que a operação em RAM.

### 3. Custo de Armazenamento
- Armazenar dados é barato, mas consultar dados é caro.
- Importância de entender o comportamento de leitura e escrita do software.
  
## Comparação entre RAM e Disco
- A RAM é mais rápida que o disco, mas existem cenários onde sua organização e tempo de resposta podem ser limitados.
- Uso de **Redis** e **Memcached** para acelerar o acesso à memória, mas nem sempre são adequados para todos os cenários.

## Considerações Finais
- Importância de entender que **memória é lenta** em certos contextos.
- Necessidade de organizar dados de forma eficiente na memória.
- Cuidado na escolha de soluções de software que encapsulam comportamentos de armazenamento.

# Resumo da Aula sobre Design de Sistemas

## Introdução
- A aula aborda a importância de considerar recursos finitos e a complexidade na arquitetura de sistemas.
- Os principais recursos discutidos incluem memória, rede e armazenamento.

## Recursos Finitos
- É fundamental entender que:
  - **Memória é cara** e finita.
  - A **armazenagem inteligente** é necessária para otimizar o uso de recursos.

## Complexidade da Rede
- A rede é um recurso crítico que deve ser tratado com cuidado.
- **Latência** é um dos principais problemas a ser considerado.
  - Maior número de **round trips** resulta em maior latência.
  - Exemplo: Transferir 1 kilobyte em uma rede de 1 gigabit por segundo pode levar 10 microsegundos, mas em um cenário com múltiplos round trips, esse tempo pode aumentar significativamente (ex: 500 microsegundos).

### Impacto da Distribuição Geográfica
- A latência aumenta em cenários onde os recursos estão distribuídos em diferentes regiões.
- **Condições de ambiente** são cruciais para o desempenho da rede.

## Armazenamento
- O armazenamento é um dos fatores mais importantes para o tempo de resposta de aplicações.
- **Discos HDD** (Hard Disk Drives) são considerados um "calcanhar de Aquiles":
  - A leitura e a escrita nos HDDs são limitadas por sua velocidade (ex: 7.200 RPM).
  - O armazenamento é barato, mas consultar dados é caro.

### Comparação entre Armazenamento e Memória
- A memória RAM é mais rápida que os discos, mas ainda pode ser considerada lenta em alguns contextos.
- A organização eficiente da memória é crucial para evitar latência em acessos.

## Comprimibilidade dos Dados
- Para um uso correto da rede, **dados devem ser comprimíveis**.
- A compressão é uma premissa importante ao desenvolver sistemas.

## Considerações sobre Arquitetura
- A arquitetura deve sempre considerar:
  - Como os dados são armazenados e o custo associado a esse armazenamento.
  - A necessidade de otimização do uso da memória e das operações de leitura/escrita.

### Cenários Específicos
- Em algumas situações, como o uso de leitura biométrica, pode ser necessário gerenciar a memória de maneira mais direta e não depender de soluções pré-definidas como Redis ou Memcached.

## Importância do Ambiente
- O ambiente onde o software opera pode afetar significativamente o desempenho.
- A escolha da arquitetura deve ser feita com base nas necessidades específicas do projeto.

## Conclusão
- O desenvolvimento de software e design de sistemas requer uma compreensão profunda dos recursos disponíveis e de suas limitações.
- A abordagem cuidadosa em relação à rede e armazenamento pode impactar diretamente o sucesso do sistema.

## Perguntas e Discussões
- Oportunidade para os participantes questionarem e discutirem temas abordados.
- Importância do feedback e questionamentos para o aprimoramento contínuo do design de sistemas. 

### Dicas de Estudo
- Revise os conceitos de latência, compressão e armazenamento em diferentes tipos de discos.
- Pratique a comparação entre diferentes soluções de armazenamento e suas implicações de desempenho.
- Discuta casos práticos de como o ambiente pode afetar a arquitetura de um sistema.

# Estudo sobre Disaster Recovery (DR)

## O que é DR?
- **Definição**: *Disaster Recovery* (Recuperação de Desastres) é um cenário de redundância.
- **Objetivo**: Criar um ambiente resiliente a falhas, permitindo que a operação continue sem interrupções para os usuários.

## Cenários de Uso
- **Setor Bancário**:
  - Necessidade de DR em casos críticos.
  - Importância de redirecionar usuários para um ambiente de recuperação.

## Ambientes Multizonais
- **Conceito**: Utilização de ambientes em várias regiões ou zonas.
- **Benefício**: Caso uma zona falhe, é possível redirecionar usuários para outra zona, mantendo a operação.

## Métricas de DR
### RTO e RPO
- **RTO (Recovery Time Objective)**: Tempo desejado para recuperação após uma falha.
- **RPO (Recovery Point Objective)**: Tempo máximo aceitável de perda de dados.
- Importância das métricas:
  - Auxiliam na definição do número de servidores pré-provisionados (N + X).
  - Quanto mais críticos os dados, menor deve ser o RTO e o RPO.

## Pré-Provisionamento
- **Necessidade**: Em cenários críticos, provisionar servidores antecipadamente é crucial.
- **Cenário crítico**:
  - É necessário ter servidores prontos para evitar perda de vendas ou interrupções significativas.
  - Em situações onde a velocidade de recuperação é vital, o provisionamento deve ser feito para suportar uma quantidade significativa de usuários.

## Elasticidade e Escalonamento
- **Crescimento Elástico**: Capacidade de aumentar a quantidade de servidores conforme a demanda.
- **Escalonamento Horizontal**: Aumento gradual de servidores durante a transferência de workloads para o DR.

## Considerações Finais
- A relação entre a criticidade do ambiente e a necessidade de recursos pré-provisionados é direta:
  - Quanto mais crítico o ambiente, mais servidores extras devem ser preparados.
  - Em ambientes menos críticos, o tempo de recuperação pode ser mais flexível (ex: uma hora de tolerância).

## Resumo das Variáveis
- **RPO e RTO**: Determinam a quantidade de recursos necessários.
- **Desidratação**: Refere-se à capacidade de suportar usuários durante uma falha. Um DR "desidratado" pode suportar apenas uma fração dos usuários.

## Interatividade
- O palestrante encoraja perguntas e feedback dos participantes para garantir a compreensão do assunto.

---

Esses pontos fornecem uma visão abrangente sobre a importância do Disaster Recovery, suas métricas e a necessidade de um planejamento adequado para garantir a continuidade dos negócios em situações de falha.

# Estudo sobre Disaster Recovery (DR)

## O que é DR?
- **Definição**: *Disaster Recovery* (DR) refere-se a um cenário que permite a continuidade de operações em caso de falhas, garantindo uma experiência ininterrupta para os usuários.
- **Objetivo**: Criar um ambiente resiliente a falhas, onde a operação continua mesmo após uma interrupção.

## Importância do DR em Cenários Críticos
- Exemplo: Em um banco, é essencial ter um DR para redirecionar usuários em caso de falha crítica.
- **Multizonas**: Utilização de serviços em múltiplas zonas/regiões para maior segurança e resiliência.

## Termos Relevantes
- **RTO (Recovery Time Objective)**: Tempo máximo tolerável para a recuperação após uma interrupção.
- **RPO (Recovery Point Objective)**: Pontos de dados que podem ser perdidos em uma falha; define a quantidade de dados que podem ser perdidos antes da recuperação.
  
### Perguntas Frequentes
- **Como minimizar RTO e RPO?**
  - Preparar um ambiente com *N mais X* servidores pré-provisionados para garantir recuperação rápida.
  
## Estratégias de Implementação de DR
1. **Provisionamento Prévio**:
   - Ter máquinas e recursos já provisionados para evitar atrasos durante a transferência de usuários.
  
2. **Capacidade Elástica**:
   - Implementar um escalonamento horizontal que permita aumentar gradualmente a capacidade de servidores conforme necessário.
  
3. **Ambientes Desidratados**:
   - Considerar um DR "desidratado", onde apenas uma parte da infraestrutura está disponível inicialmente, mas pode ser expandida conforme a demanda.
  
## Considerações Críticas
- **Ambientes Críticos vs. Não Críticos**:
  - Para ambientes críticos, a necessidade de um DR menos desidratado é maior, exigindo mais recursos pré-provisionados.
  - Em ambientes não críticos, o tempo de recuperação pode ser mais flexível, permitindo tolerância a falhas.

## Conclusão
- A configuração de um DR eficaz depende da análise cuidadosa dos RTO e RPO, bem como da criticidade do ambiente operacional.
- Com a infraestrutura como código, como o Terraform, é possível provisionar rapidamente recursos em caso de falhas, aumentando a agilidade do DR.

# Notas de Estudo sobre Disaster Recovery (DR)

## O que é Disaster Recovery (DR)?
- **Definição**: *Disaster Recovery* é um conjunto de práticas que visa garantir a continuidade das operações de uma empresa em caso de falhas ou desastres.
- **Objetivo**: Criar um ambiente de redundância que permita a recuperação rápida e eficiente, sem que os usuários percebam a falha.

## Importância do DR
- **Cenários Críticos**: Essencial em setores como bancos, onde a continuidade das operações é crucial.
- **Redundância**: Permite redirecionar usuários para um ambiente alternativo em caso de falha.

## Conceitos Chave
### RTO e RPO
- **RTO (Recovery Time Objective)**: Tempo máximo aceitável para a recuperação após uma falha.
- **RPO (Recovery Point Objective)**: Tempo máximo aceitável de perda de dados após um desastre.
- **Importância**: Compreender esses parâmetros é vital para planejar a infraestrutura de DR.

### Ambientes Multizonais
- **Estratégia**: Utilização de múltiplas zonas ou regiões geográficas para aumentar a resiliência.
- **Benefício**: Se uma zona falhar, os usuários podem ser redirecionados para outra zona, minimizando o impacto.

## Estrutura de DR
- **Pré-Provisionamento**: Ter recursos (servidores, máquinas) já provisionados antes de uma falha.
  - *Exemplo*: Ter N+2 ou N+X servidores prontos para evitar atrasos na recuperação.
  
- **Desidratação de DR**:
  - **Definição**: Operar com um DR que não suporta 100% dos usuários de uma vez.
  - **Estratégia**: Suportar uma porcentagem (ex: 30%) inicialmente e aumentar conforme necessário.
  - **Crescimento Elástico**: Capacidade de ampliar recursos horizontalmente à medida que a demanda aumenta.

## Considerações Finais
- **Sensibilidade das Métricas**: Quanto mais crítico o ambiente, mais recursos devem ser pré-provisionados.
- **Implementação Rápida**: Com ferramentas como *Terraform*, o provisionamento de infraestrutura pode ser feito rapidamente.
- **Avaliação Contínua**: A operação deve ser constantemente avaliada para ajustar RTO e RPO conforme a criticidade do negócio.

## Resumo das Perguntas Frequentes
1. **O que é DR?**
   - Cenário de redundância para continuidade de operações.
   
2. **Qual a importância de RTO e RPO?**
   - Definem os limites de tempo para recuperação e perda de dados.
   
3. **Como funciona a estratégia multizonal?**
   - Permite redirecionar usuários para diferentes zonas em caso de falha.

4. **O que significa ter um DR desidratado?**
   - Operar com apenas uma parte da capacidade total de recuperação.

## Conclusão
A implementação de um bom plano de *Disaster Recovery* é fundamental para garantir a continuidade das operações de negócios em situações adversas. Compreender os conceitos de RTO, RPO e a estrutura necessária é essencial para o sucesso nesta área.

## Desidratação do DR

- **Conceito de desidratação**: Refere-se à capacidade do DR de suportar a carga de usuários durante uma falha.
- **Percentual de suporte**: Pode-se planejar suportar inicialmente uma fração (ex: 30%) dos usuários e aumentar gradativamente.

## Considerações Finais

- **Críticos vs. Não Críticos**: A necessidade de recursos extras depende do nível crítico do ambiente.
  - Ambientes não críticos podem tolerar tempos de inatividade mais longos.
- **Provisionamento em nuvem**: 
  - Ferramentas como *Terraform* podem acelerar o processo de provisionamento de infraestrutura.
  - Em ambientes de nuvem, máquinas podem ser provisionadas rapidamente, tornando a recuperação mais eficiente.

## O que é System Design?
- O *System Design* é um processo de definição de:
  - Arquitetura
  - Componentes
  - Módulos
  - Interfaces
  - Dados
- Objetivo: atender requisitos especificados.

### Importância do System Design
- O *System Design* vai além de apenas desenhar a arquitetura:
  - **Intencionalidade**: Ajuda a pensar de forma intencional sobre o problema a ser resolvido.
  - **Racionalização**: Facilita a definição das partes realmente importantes do sistema.
  - **Aproximações**: Focar em aproximações é mais eficaz do que buscar soluções extremamente assertivas.

## Etapas do System Design
- **Planejamento de Soluções**: Explorar diversas soluções para um problema específico.
- **Implicações Futuras**: Considerar como as decisões de design afetarão o sistema a longo prazo (ex. custos de armazenamento).
- **Exercício Mental**: Desenvolver a capacidade de planejar e avaliar diferentes abordagens.

## Exemplos de Decisões em System Design
- **Performance vs. Custo**:
  - Uso de *cache* em memória pode melhorar a performance, mas é caro.
  - Avaliar alternativas que possam ser mais econômicas dependendo do contexto.

## Conclusão
- O *System Design* é um processo crítico que envolve a definição cuidadosa de sistemas, levando em consideração a arquitetura, os requisitos e as implicações futuras. A prática e o entendimento profundo dos conceitos são essenciais para a criação de soluções eficazes.
